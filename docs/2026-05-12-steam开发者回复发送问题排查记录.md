# Steam 开发者回复发送问题排查记录

本文记录 2026-05-12 对 Steam 开发者回复发送链路的一次完整排查过程，目标是说明：

- 我们实际遇到了哪些现象
- 每种现象分别对应什么问题
- 最终验证通过的服务器配置是什么
- 后续再次出现类似问题时，应该按什么顺序排查

## 一、问题背景

系统支持在后台审核 AI 草稿后，调用 Steam Community 的开发者回复接口发送评论回复：

```text
POST https://steamcommunity.com/userreviews/setdeveloperresponse/{recommendationid}
```

发送链路依赖三部分：

- 有效的 Steam 登录 Cookie
- 可访问 `steamcommunity.com` 的网络出口
- 后端发送状态与错误信息的正确回写

这次排查过程中，这三部分都分别出现过问题。

## 二、最初现象

### 1. 点击发送后，草稿先从待审核列表消失

表现：

- 前端点击“通过发送”后，草稿立刻不在待审核列表里
- 页面没有明确显示“发送中”
- 过一会刷新后才看到“发送失败”

原因：

- 旧逻辑会把草稿状态先改成 `sending`
- 但待审核列表接口最初没有把 `sending` 状态返回给前端
- 所以前端会误以为这条记录消失了

处理结果：

- `sending` 状态保留在待审核列表中
- 前端显示“发送中”
- 发送失败时会直接显示错误信息

### 2. API 返回 202，但最终发送失败

表现：

- 接口 `POST /api/reviews/{id}/send-reply` 返回 `202 Accepted`
- 前端没有同步拿到 Steam 的最终结果
- Docker 日志初期信息不足

原因：

- 发送是后台异步执行，不是在 API 请求内同步完成
- `202` 只表示“已进入发送队列”，不表示 Steam 已发送成功

处理结果：

- 后端加强了失败日志
- 失败原因会回写到 `developer_replies.error_message`
- 前端会显示更明确的错误信息

## 三、排查过程中遇到的三类真实问题

这次不是单一问题，而是连续碰到了三类不同问题。

### 问题 A：服务器无法访问 `steamcommunity.com`

最早的服务器日志是：

```text
httpx.ConnectTimeout
```

对应现象：

- `store.steampowered.com` 能访问
- `steamcommunity.com` 超时
- 发送链路失败

宿主机上实际验证：

```bash
curl -I --max-time 20 https://steamcommunity.com/
curl -I --max-time 20 https://store.steampowered.com/
getent hosts steamcommunity.com
```

当时结论是：

- 服务器直连 `steamcommunity.com` 不稳定或不可达
- 发送失败不是因为文案，而是网络连不上 Community 域名

### 问题 B：本地非 Docker 运行时，Cookie 文件路径错误

本地日志里出现过：

```text
Cookie file does not exist: /app/data/steam_cookie.txt
```

原因：

- `.env` 里的 `STEAM_COOKIE_FILE` 使用了 Docker 容器路径 `/app/data/steam_cookie.txt`
- 本地直接运行后端时并不存在 `/app/...`

处理结果：

- 后端已兼容这种情况
- 本地非 Docker 运行时，如果配置的是 `/app/data/steam_cookie.txt`，会自动映射到仓库根目录下的 `data/steam_cookie.txt`

### 问题 C：请求打到了 Steam，但被 Steam 业务层拒绝

本地日志里还出现过：

```text
{"recommendationid": "224757388", "success": false, "response": {"success": 15}}
```

这说明：

- 请求已经成功到达 Steam
- 不是网络超时
- 不是 DNS 问题
- 是 Steam 自己返回了业务失败

这类错误通常优先怀疑：

- 当前 Cookie 对应账号不是该 App 的开发者/发行方账号
- 账号登录态和当前出口网络不匹配
- 浏览器成功时用的是另一条代理出口，而程序请求走了不同出口

## 四、关键结论：发送链路和抓取链路不能混为一谈

这次排查的一个核心结论是：

- 抓取评论主要依赖 `store.steampowered.com`
- 发送开发者回复必须访问 `steamcommunity.com`

两者不是同一个网络需求。

因此：

- 抓取代理能用，不代表发送代理也能用
- 海外代理能打开 Store，不代表也能稳定写 Community

## 五、Oxylabs 代理的最终验证结果

后续在 `backend` 容器里对 Oxylabs 做了单独测试，结论如下：

- `steamcommunity.com` 通过 **Oxylabs HTTPS 代理** 可以访问
- `steamcommunity.com` 通过 **Oxylabs HTTP 代理** 会报 `ReadError`
- `store.steampowered.com` 通过两者通常都可访问

也就是说，Steam Community 对 Oxylabs 的使用方式更敏感，实际可用的是：

```env
OXYLABS_PROXY_SCHEME=https
```

不是：

```env
OXYLABS_PROXY_SCHEME=http
```

## 六、发送代理配置的最终方案

项目现在支持两种发送模式。

### 模式 1：发送走独立代理

适合本地 Clash、VPN、本地 HTTP 代理，或服务器单独可用的出口代理。

使用环境变量：

```env
STEAM_REPLY_PROXY_URL=http://127.0.0.1:7890
STEAM_REPLY_PROXY_DIRECT_FALLBACK=false
```

说明：

- 配置了 `STEAM_REPLY_PROXY_URL` 后，发送优先走这条代理
- 这条代理和 Oxylabs 抓取代理完全独立

### 模式 2：发送继续走 Oxylabs

如果不配置 `STEAM_REPLY_PROXY_URL`，发送会回退到旧的 Oxylabs 发送逻辑。

服务器这次最终验证通过的配置建议是：

```env
STEAM_REPLY_PROXY_URL=
STEAM_REPLY_PROXY_DIRECT_FALLBACK=false
OXYLABS_PROXY_ENABLED=true
OXYLABS_PROXY_SCHEME=https
OXYLABS_PROXY_DIRECT_FALLBACK=false
```

说明：

- `STEAM_REPLY_PROXY_URL` 留空，表示发送不走独立代理
- 发送改为走 Oxylabs
- `OXYLABS_PROXY_SCHEME=https` 是关键
- 关闭 direct fallback，避免代理失败后又回退直连，导致日志被误导成单纯的 `ConnectTimeout`

改完 `.env` 后需要重建后端容器：

```bash
docker-compose up -d --build --force-recreate backend
```

## 七、浏览器成功但程序失败的判断方法

这次还验证了一个很重要的事实：

- 浏览器里手工发送成功
- 程序发送却失败

浏览器抓包显示：

- 请求就是发往 `https://steamcommunity.com/userreviews/setdeveloperresponse/{recommendationid}`
- 浏览器当时走的是本地代理 `127.0.0.1:7890`

这意味着：

- Cookie 是否有效，不能脱离“出口网络”单独看
- 同一份 Cookie 在不同出口上，Steam 的行为可能不一样

实务上要这样判断：

- 浏览器成功 + 程序失败：优先查代理出口是否一致
- 浏览器失败 + 程序失败：优先查 Cookie 是否过期或账号权限是否不足

## 八、推荐排查顺序

后续再遇到“点击发送失败”，按下面顺序查，不要跳步。

### 第一步：看后端日志是哪一类错误

重点看三类：

- `ConnectTimeout`
- `Cookie file does not exist`
- `success: false` 且 `response.success != 1`

它们分别代表：

- 网络不可达
- Cookie 文件路径问题
- Steam 业务层拒绝

### 第二步：先判断是宿主机问题还是容器问题

宿主机检查：

```bash
curl -I --max-time 20 https://steamcommunity.com/
curl -I --max-time 20 https://store.steampowered.com/
getent hosts steamcommunity.com
```

容器检查：

```bash
docker-compose exec -T backend sh -lc 'cd /app && python - <<'"'"'PY'"'"'
import socket

print("getaddrinfo for steamcommunity.com:")
for item in socket.getaddrinfo("steamcommunity.com", 443, proto=socket.IPPROTO_TCP):
    family, _, _, _, sockaddr = item
    fam = "IPv6" if family == socket.AF_INET6 else "IPv4"
    print(fam, sockaddr)
PY'
```

### 第三步：确认容器里真实拿到了什么代理配置

```bash
docker-compose exec backend sh -lc 'env | grep OXYLABS | sort'
docker-compose exec backend sh -lc 'python - <<'"'"'PY'"'"'
from src.utils.oxylabs_proxy import load_oxylabs_proxy_settings
print(load_oxylabs_proxy_settings())
PY'
```

如果使用独立发送代理，还要确认：

```bash
docker-compose exec backend sh -lc 'env | grep STEAM_REPLY_PROXY'
```

### 第四步：直接在容器里测代理能不能访问 Community

下面这段命令不发送回复，只测网络可达性：

```bash
docker-compose exec -T backend sh -lc 'cd /app && python - <<'"'"'PY'"'"'
import asyncio
import httpx
from src.utils.oxylabs_proxy import load_oxylabs_proxy_settings

settings = load_oxylabs_proxy_settings()
print("enabled =", settings.enabled)
print("host =", settings.host)
print("scheme =", settings.scheme)
print("direct_fallback =", settings.direct_fallback)

async def test(label, proxy, url):
    print(f"\n== {label} ==")
    try:
        async with httpx.AsyncClient(
            proxy=proxy,
            timeout=httpx.Timeout(20.0, connect=20.0),
            verify=False,
            follow_redirects=True,
            trust_env=False,
        ) as client:
            r = await client.get(url)
            print("status =", r.status_code)
            print("final_url =", str(r.url))
            print("server =", r.headers.get("server"))
    except Exception as e:
        print(type(e).__name__, str(e))

async def main():
    if not settings.enabled:
        print("Oxylabs is disabled")
        return

    urls = [
        ("steamcommunity", "https://steamcommunity.com/"),
        ("store", "https://store.steampowered.com/"),
        ("oxylabs-location", "https://ip.oxylabs.io/location"),
    ]

    sticky_port = settings.choose_sticky_session_port()
    for label, url in urls:
        for scheme in ("https", "http"):
            proxy = settings.build_sticky_proxy_url(
                session_port=sticky_port,
                scheme_override=scheme,
            )
            await test(f"sticky {scheme} {label}", proxy, url)

asyncio.run(main())
PY'
```

如果结果是：

- `https steamcommunity -> 200`
- `http steamcommunity -> ReadError`

那就说明 Oxylabs 能用，但只能按 HTTPS 入口去用。

### 第五步：确认是否是 Cookie/登录态问题

如果网络已经通，但发送接口返回：

```text
success: false
```

或者：

```text
response.success = 15
```

优先这样查：

1. 用同一份 Cookie 对应的账号，在浏览器里手工打开对应评论页面
2. 确认该账号能不能手工发送开发者回复
3. 确认浏览器成功时走的出口网络，是否和程序发送时一致

如果浏览器能发，程序不能发，重点不是文案，而是：

- 请求出口不同
- Cookie 会话和当前出口不匹配

## 九、这次排查后的代码改动摘要

本次排查相关改动主要包括：

- 待审核列表保留 `sending` 状态，不再点击后直接消失
- 发送失败信息会回写到草稿和回复记录
- 发送异常会输出更明确的后端日志
- 本地非 Docker 运行时兼容 `/app/data/steam_cookie.txt`
- 发送链路支持独立代理 `STEAM_REPLY_PROXY_URL`
- 任务页能显示发送链路实际使用的代理状态

## 十、当前推荐结论

如果服务器已经实测“可以发送”，当前建议是：

- 优先保留当前可用配置，不要频繁切换代理方案
- 明确把发送链路和抓取链路分开看待
- 再次出问题时，先判断是网络、Cookie 路径、还是 Steam 业务拒绝

一句话总结：

> Steam Store 可访问，不代表 Steam Community 可发送；发送成功依赖 Cookie、账号权限和出口网络三者同时成立。
