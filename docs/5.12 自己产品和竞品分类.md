# 自家游戏与竞品分离改造计划

## Summary

把当前“所有游戏一套能力”的模型改成二分类：`自家游戏` 与 `竞品`。  
目标是让系统默认服务自家游戏运营，竞品只保留情报查看和导出，不再暴露草稿生成、审核、发送回复等操作链路。

已确认的决策：

- 游戏分类采用二分类
- 界面采用“双视图切换”
- 首页总览默认汇总全部自家游戏，并支持切到单个自家游戏
- 现有数据回填：`3350200` 标记为自家，其余现有游戏默认竞品

## Key Changes

### 1. 数据模型与回填

- 在 `steam_games` 增加分类字段，建议命名为 `game_scope`，取值固定为 `owned` / `competitor`
- 新增 Alembic 迁移：
  - 给字段加非空默认值
  - 回填 `3350200 -> owned`
  - 其余现有游戏回填为 `competitor`
- 游戏管理接口和前端游戏列表增加该字段，可查看、编辑分类
- 新建游戏时默认分类为 `competitor`，必须显式切为 `owned` 才启用自家运营能力

### 2. 总览与统计口径

- `/api/stats/overview` 和 `/api/stats/timeseries` 增加游戏范围参数：
  - 默认 `scope=owned`
  - 可选 `app_id`，仅允许选自家游戏做单游戏总览
- 首页默认展示“全部自家游戏汇总”
- 首页增加自家游戏切换器：
  - `全部自家`
  - 单个 `owned` 游戏
- 竞品不进入首页总览统计口径
- 现有统计定义保持不变，只改变数据范围

### 3. 评论列表拆分与权限边界

- 评论列表页改成双视图：
  - `自家游戏`
  - `竞品`
- 默认进入 `自家游戏` 视图，且默认仍显示“待处理”分组
- `自家游戏` 视图：
  - 保留当前的筛选、详情、草稿生成、审核、发送、导出
- `竞品` 视图：
  - 仅保留查看、筛选、详情、导出
  - 隐藏或禁用所有会改变回复状态的入口：
    - 生成草稿
    - 批量生成草稿
    - 发送回复
    - 驳回/保存草稿
    - 删除需求记录
- 后端也要做硬限制，不能只靠前端隐藏：
  - 回复生成、发送、草稿编辑相关路由在 `competitor` 游戏上直接拒绝
  - 返回明确错误，例如“竞品游戏不支持回复操作”

### 4. 回复记录与审核页范围收口

- 回复记录页、待审核草稿页只展示 `owned` 游戏
- 竞品不进入回复记录、审核队列、发送记录链路
- 如果用户通过旧链接或直接请求竞品回复记录接口，后端返回空结果或明确拒绝
- 当前按处理时间分组的逻辑保持不变，只改变可见游戏范围

### 5. 导出与接口调整

- 评论导出 Excel 保持两种模式：
  - `导出当前`
  - `导出全部`
- 但导出范围跟随当前视图：
  - 在 `自家游戏` 视图导出自家数据
  - 在 `竞品` 视图导出竞品数据
- 公开接口/类型需要增加或调整：
  - `SteamGame` / `GameListItem` / `GameCreateRequest` / `GameUpdateRequest` 增加 `game_scope`
  - `stats` 接口增加 `scope` 和可选 `app_id`
  - `reviews` 列表接口增加 `game_scope` 或等效视图参数
- 回复相关接口不需要新增参数，但要在服务层校验 `review.app_id -> game_scope`

## Test Plan

- 数据迁移：
  - 迁移后 `steam_games.game_scope` 非空
  - `3350200` 为 `owned`
  - 其他现有游戏为 `competitor`
- 首页：
  - 默认只统计 `owned`
  - 切单个自家游戏时统计正确
  - 竞品不会进入首页总览
- 评论列表：
  - 默认进入自家视图 + 待处理分组
  - 切到竞品视图后仍可筛选、查看详情、导出
  - 竞品视图不显示回复相关按钮
- 后端权限：
  - 对竞品评论调用生成草稿、发送回复、草稿修改接口时返回拒绝
  - 自家游戏链路保持可用
- 回复记录：
  - 只出现 `owned` 游戏
  - 时间分组继续按处理时间，不回退到帖子发布时间
- 导出：
  - 自家视图和竞品视图都能导出
  - `导出当前` 与 `导出全部` 范围正确
- 回归：
  - 已回复评论不会再显示为待处理
  - 回复发送成功后状态仍会更新为 `completed + replied`

## Assumptions

- `owned` 是唯一允许进入回复运营链路的游戏类型
- `competitor` 只读，不允许任何回复相关写操作
- 首页“单游戏总览”只支持自家游戏，不支持竞品
- 新增游戏默认按 `competitor` 处理，避免误开放回复能力
- 这次不做更复杂的多分类体系，也不做基于角色的细粒度竞品权限差异
