# 多游戏监控与按游戏切换评论的改造方案

## 摘要
把当前“单个游戏、单条同步配置”的实现改成“多个游戏监控任务 + 评论列表按游戏切换 + 任务队列展示每个监控任务的执行记录”。

本次默认决策：
- 监控任务支持 **多个**
- 每个监控任务固定绑定 **一个 `app_id`**
- 定时策略只做 **每天几点执行**
- **不做分钟选择**，后端统一按该小时的 `00` 分触发
- 评论列表顶部增加 **游戏切换器**
- 任务队列展示 **监控任务信息 + 实际执行记录**
- 手动同步仍保留，但手动同步也要求先选中一个具体游戏

## 关键改造

### 1. 数据模型与调度行为
- 将 `task_schedules` 从“唯一一条 `steam_review_sync` 配置”改成“多条监控配置”：
  - 新增 `name` 字段，作为监控任务名称
  - 保留 `task_type`，固定为 `steam_review_sync`
  - 保留 `app_id`、`is_enabled`、`options`
  - `interval` 固定支持 `daily`
  - `hour` 为必填，`0-23`
  - `minute` 后端固定存 `0`，前端不暴露
- 给 `sync_jobs` 增加可空 `schedule_id`，关联到具体监控任务：
  - 定时触发的执行记录写入对应 `schedule_id`
  - 手动同步允许 `schedule_id = null`
- 调度器每轮扫描所有启用的监控任务：
  - 到达配置小时且当天未执行过，则创建一条新的 `sync_jobs`
  - 同一 `schedule_id` 若已有 `pending/running` 任务，则跳过
  - 同一 `app_id` 允许存在多个监控任务，但同一监控任务每天只跑一次
- 评论同步逻辑继续沿用当前“按本地最新评论时间追新直到无更新”的实现，不改抓取策略

### 2. 后端接口与返回结构
- 将当前单条配置接口改为面向列表：
  - `GET /api/tasks/schedules`：返回全部监控任务
  - `POST /api/tasks/schedules`：创建监控任务
  - `PATCH /api/tasks/schedules/{id}`：更新监控任务
  - `DELETE /api/tasks/schedules/{id}`：删除监控任务
- 保留手动同步入口，但请求体支持可选 `schedule_id`，用于把手动执行与某个监控任务挂钩
- `GET /api/tasks` 与 `GET /api/tasks/{task_id}` 的返回增加：
  - `schedule_id`
  - `schedule_name`
  - `trigger_type`（`manual` / `scheduled`）
- 评论列表接口继续使用现有 `app_id` 过滤参数；前端切换游戏时只是在现有查询上切换 `app_id`

### 3. 前端页面改造
- **评论列表页**
  - 顶部增加“游戏切换”下拉框，数据来源为 `steam_games` 中已有评论的游戏列表
  - 默认选择第一个有评论的游戏；如果 URL 或本地状态里已有 `app_id`，优先恢复
  - 所有筛选、统计、详情、批量操作都跟随当前选中游戏
- **任务与同步页**
  - 从“单条配置表单”改成“两段式界面”：
    - 上部：监控任务列表（任务名、App ID、每天几点、启用状态）
    - 下部/弹层：创建或编辑监控任务
  - 创建任务时必须填写：
    - 任务名称
    - App ID
    - 执行小时
    - 是否启用
  - 可编辑高级同步参数，继续落到 `options`
  - 手动同步按钮改成“对当前选中的监控任务立即执行”
- **任务队列页**
  - 任务列表与详情中展示：
    - 监控任务名称
    - App ID
    - 触发方式（手动 / 定时）
    - 执行时间
  - 支持按监控任务筛选任务记录
  - 失败日志继续展示完整错误信息和 traceback

### 4. 迁移与兼容
- 新增 Alembic 迁移：
  - `task_schedules` 增加 `name`
  - `sync_jobs` 增加 `schedule_id`
  - 如需前端直接展示，可考虑在 `sync_jobs` 冗余 `trigger_type`
- 兼容旧数据：
  - 旧的唯一 `steam_review_sync` 配置如果存在，迁移时自动转成一条监控任务
  - 名称默认生成为 `App {app_id} 每日同步`
  - `minute` 保留数据库值但前端忽略，后续统一写 `0`
- 现有 `GET /api/tasks/schedule` / `PATCH /api/tasks/schedule` 作为过渡接口删除，不再继续维护，前后端统一切到新列表接口

## 测试与验收

### 后端测试
- 创建多个监控任务，列表接口返回正确
- 更新/删除监控任务生效
- 调度器在同一天同一小时只触发一次同一 `schedule_id`
- 不同 `schedule_id` 可分别创建执行记录
- 手动同步能绑定 `schedule_id`，也允许不绑定
- `GET /api/tasks`、`GET /api/tasks/{id}` 返回 `schedule_name`、`trigger_type`
- 旧配置迁移后仍可被调度

### 前端测试
- 评论列表可以切换不同游戏，切换后列表与筛选联动正确
- 任务与同步页可以新增、编辑、删除多个监控任务
- 创建任务时只能选小时，不能选分钟
- 手动同步会针对当前任务创建一条新的执行记录
- 任务队列能区分不同监控任务及其执行记录
- 失败任务仍能看到完整错误日志

### 验收标准
- 后台能同时维护多个游戏监控任务
- 每个监控任务能单独设置“每天几点执行”
- 评论列表能稳定切换不同游戏的数据
- 任务队列能清楚看出“哪个监控任务创建了哪次执行”
- 不再存在“只有单个全局同步配置”的交互

## 默认假设
- 定时任务精度按“小时”处理，实际触发时间为该小时 `00` 分
- 当前阶段不做每周、cron、分钟级调度
- 当前阶段不做批量创建多个 `app_id` 到一条任务；一条任务只管一个游戏
- 评论列表游戏选项来源于数据库已有评论数据对应的游戏，而不是所有 Steam 游戏
- 任务名称允许重复，但前端默认提示用户写成可辨识名称
